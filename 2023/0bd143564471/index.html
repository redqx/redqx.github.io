msf反弹shell木马分析 1.0
2023-12-27 10:34:10

背景: 第一次使用msf

杀软检测

360报毒

火绒报毒:

病毒类型: 代码混淆器(VirTool/Meterpreter.a)

分析过程

首先用IDA静态分析

一来就出现了很多垃圾字节码

那长长的一条全是垃圾代码

特点:

跳过垃圾代码还是很难分析,,因为在正常代码中,仍然插入了很多垃圾代码

类似于这种

于是尝试动态分析: 火绒剑+api monitor

火绒剑: 看不到关键api调用的函数栈,所以找不到哪里调用了什么

api monitor: 看不到当前模块的调用动作,,全是其它模块

比如在利用火绒剑分析的时候,,分析tcp连接的时候,,,函数的返回根本就没有用户模块,,

而且很多关键的操作甚至没有函数栈

比如在利用api monitor分析时候,比如木马是app.exe

api monitor的监控过程既然没有app.exe模块,,,

简直是有点震惊哟….

在某一个时刻,,我就疑惑

msf每次生成的木马,会不会是一样的,,,

于是尝试去生成,,,发现并不一样

在某一刻我对木马的导入表产生兴趣

发现他豪不避讳的引入了很多dll和很多API

在某一刻,,我去动态调试木马,,,

比如在LoadLibrary和GetProAddress下断点

发现LoadLibrary只断下一次,,然后参数是某个网络通信的dll文件,,,既感到疑惑,,,也没有感到疑惑

然后GetProAddress并没有断下来

之前我在api monitor分析的时候,,因为找不到本模块的API调用,,并且在一些APi的返回地址发现不在本模块

怀疑是远程线程注入什么的,,,后来发现并不是的

因为木马本身加载了很多API,,比如网络通信的API,,所以不太可能是远处线程注入

并且导入表也没有发下敏感的 有关注入的API,,,,

在某次kernel32_LoadLibraryA下断的时候

发现参数是 ws2_32

然后返回地址是

其中返回地址指向了 1D00AA

于是跟进,然后不断的C字节码

发现返回到一块debug045节区

这种节区一般是动态内存分配的,,,

所以….为什么会出现动态内存分配

估计是把一块shellcode分配到内存中,,然后分配权限,,然他执行

这和shellcode加载岂不是一会事情,,,,

我不由得想起了之前msf生成shellcode

msf生成exe和生成shellcode区别在哪

然后发现木马的exe中存在和msf生成的shellcode一模一样

所以木马只是一个shellcode加载器,,,

msf生成的shellcode如下

通过定位,,,在木马中找打了shellcode,和msf的c语言shellcode简直一模一样

在shellco1de开始那里下断点,,,,

然后在loadlibrary和getproaddress下断

发现引入了ws2_32.dll 然后返回地址不是shellcode部分

而是内存,,,

木马木马中已经具备了直接运行shellcode的条件

shellcode并没有让自己运行在木马已经有的空间中

而是分配了内容,,,然后让他执行,,,,

期间我也在getproaddress下断,,但并没有触发,,,说明了shellcode自己实现了getproaddress

而不让我们去监视,,,,为什么可以监视到loadlibrary,,因为这个东西没办法自己实现

shellcode顶多实现getmodulehandle
